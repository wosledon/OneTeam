# PR 代码评审

## 输入

- **PR URL 或编号**：`{{$PR_ID}}`
- **评审重点**（可选）：`{{$FOCUS}}`

## 评审维度

对 PR 的以下维度逐一检查并给出结论：

| 维度           | 评审要点                                           |
| -------------- | -------------------------------------------------- |
| **需求覆盖**   | 所有需求点是否都有对应实现？是否有遗漏或多余？     |
| **逻辑正确性** | 业务逻辑是否正确？边界条件、异常流程是否处理完整？ |
| **代码质量**   | 命名清晰、职责单一、无重复代码、无隐式依赖         |
| **安全性**     | SQL 注入、XSS、越权、敏感信息泄露是否防范？        |
| **性能**       | 是否有 N+1 查询、不必要的循环、内存泄漏风险？      |
| **测试**       | 关键逻辑是否有测试覆盖？测试质量如何？             |
| **可维护性**   | 对后续修改是否友好？是否引入不必要的复杂度？       |

## 输出格式

```
## PR 信息
- PR 编号/URL
- 改动规模（文件数、行数）

## 评审结论
[通过 / 有条件通过 / 拒绝]

## 问题清单
### 🔴 Blocking（必须修复）
### 🟡 Non-blocking（建议修复）
### 💬 nit（可以忽略）

## 风险评级
- [ ] 低风险
- [ ] 中风险
- [ ] 高风险

## 总结建议
```

## 评审原则

- 只对 PR 内容负责，不重新实现
- 问题必须有具体位置（文件:行号），禁止模糊描述
- 区分 Blocking / Non-blocking / nit三级，不混用
- 如果发现未关联需求的改动，必须标注并要求说明
